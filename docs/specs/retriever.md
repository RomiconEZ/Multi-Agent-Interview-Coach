# Spec: Retriever (Context Retrieval)

В текущей архитектуре Multi-Agent Interview Coach **отсутствует классический RAG-контур** (нет векторной базы, нет индексов, нет семантического поиска по документам). Retrieval-подобные механизмы реализованы как детерминированная логика формирования контекста для каждого агента.

---

## 1. Источники данных

| Источник | Тип | Хранилище | Описание |
|---|---|---|---|
| История диалога | Structured (Pydantic) | In-memory (`InterviewState.turns`) | Полная история `InterviewTurn` за сессию: `agent_visible_message`, `user_message`, `internal_thoughts`, `timestamp` |
| Информация о кандидате | Structured (Pydantic) | In-memory (`InterviewState.candidate`) | `CandidateInfo`: name, position, target_grade, experience, technologies. Извлекается Observer из текста кандидата |
| Навыки и пробелы | Structured (list) | In-memory (`InterviewState`) | `confirmed_skills: list[str]`, `knowledge_gaps: list[dict]`, `covered_topics: list[str]` |
| Описание вакансии | Unstructured (str) | In-memory (`InterviewState.job_description`) | Опциональный текст, введённый пользователем при старте сессии |
| Активный якорь | Unstructured (str) | In-memory (`InterviewSession._last_agent_message`) | Последнее сообщение интервьюера — текущий открытый вопрос |
| Метрики сессии | Structured (dataclass) | In-memory (`SessionMetrics`) | Агрегированные токены по агентам, стоимость, количество ходов и генераций |

---

## 2. Стратегии формирования контекста (по агентам)

### 2.1 ObserverAgent

**Файл**: `src/app/agents/observer.py` → `_build_analysis_context()`

| Компонент контекста | Источник | Обработка | Бюджет |
|---|---|---|---|
| Информация о кандидате | `CandidateInfo` | Форматирование полей в текст | ~50–100 токенов |
| Описание вакансии | `job_description` | `_build_job_description_block()` → XML-тег `<job_description>` | ~100–500 токенов (зависит от длины) |
| Резюме истории | `InterviewState.turns[-5:]` | `_summarize_history()` — обрезка каждой реплики до 100 символов | ~200–500 токенов |
| Последний вопрос интервьюера | `_last_agent_message` | Передаётся как есть (полный текст) | ~50–200 токенов |
| Сообщение кандидата | `user_message` | Обёрнуто в `<user_input>...</user_input>` с инструкцией безопасности | ~50–500 токенов |

**Общий бюджет контекста**: ~500–1800 токенов (без системного промпта).

**Ограничения**:

- Observer получает **краткое резюме** последних 5 ходов, а не полную историю.
- Реплики обрезаются до 100 символов для экономии контекстного окна.
- При отсутствии истории возвращается строка `"Интервью только началось."`.

### 2.2 InterviewerAgent

**Файл**: `src/app/agents/interviewer.py` → `_build_response_context()`

| Компонент контекста | Источник | Обработка | Бюджет |
|---|---|---|---|
| Информация о кандидате | `CandidateInfo` | Форматирование полей + акцент на `technologies` | ~100–200 токенов |
| Описание вакансии | `job_description` | `_build_job_description_block()` → XML-тег | ~100–500 токенов |
| Текущее состояние | `InterviewState` | difficulty, confirmed_skills, knowledge_gaps count | ~50–100 токенов |
| Активный якорь | `turns[-1].agent_visible_message` | Полный текст последнего сообщения интервьюера | ~50–200 токенов |
| Сообщение кандидата | `user_message` | Обёрнуто в `<user_input>...</user_input>` | ~50–500 токенов |
| Анализ Observer | `ObserverAnalysis` | response_type, quality, is_factually_correct, is_gibberish, answered_last_question, recommendation, correct_answer | ~100–200 токенов |
| Инструкция действия | `_get_response_instruction()` | Детерминированная ветка по response_type + answered_last_question | ~50–150 токенов |
| Полная история (window) | `get_conversation_history(HISTORY_WINDOW_TURNS)` | Последние N ходов как role/content pairs | ~500–3000 токенов |

**Общий бюджет контекста**: ~1000–5000 токенов (без системного промпта).

**Ограничения**:

- Полная история ограничена `HISTORY_WINDOW_TURNS` (default: 10 ходов).
- При длинных сессиях ранние ходы выпадают из контекстного окна.
- `_build_messages()` обеспечивает корректное чередование `user`/`assistant` ролей, вставляя фиктивное сообщение при необходимости.

### 2.3 EvaluatorAgent

**Файл**: `src/app/agents/evaluator.py` → `_build_evaluation_context()`

| Компонент контекста | Источник | Обработка | Бюджет |
|---|---|---|---|
| Информация о кандидате | `CandidateInfo` | Форматирование полей | ~50–100 токенов |
| Статистика интервью | `InterviewState` | total_turns, turns_with_answers, final_difficulty | ~50–100 токенов |
| Описание вакансии | `job_description` | `_build_job_description_block()` | ~100–500 токенов |
| Полная история диалога | `InterviewState.turns` | `_format_conversation()` — **все ходы** с мыслями агентов | ~2000–8000 токенов |
| Сводка навыков | `confirmed_skills`, `knowledge_gaps`, `covered_topics` | `_format_skills_summary()` | ~100–500 токенов |

**Общий бюджет контекста**: ~2500–9200 токенов (без системного промпта).

**Ограничения**:

- Evaluator получает **полную историю** (без оконного ограничения), так как оценивает всю сессию.
- Для длинных сессий (20+ ходов) контекст может приближаться к лимиту моделей с малым окном (< 8K).
- Мысли агентов (`internal_thoughts`) также включаются в контекст Evaluator для полноты картины.

---

## 3. Reranking

Не применяется. Контекст формируется детерминированно без ранжирования фрагментов.

---

## 4. Индексирование

Не применяется. Система не строит индексы. Все данные хранятся в структурированном виде (Pydantic-модели) и извлекаются прямым доступом к полям.

---

## 5. Ограничения текущего подхода

| Ограничение | Описание | Потенциальное решение |
|---|---|---|
| Нет внешнего источника истины | Observer полагается на знания LLM для факт-чекинга. `correct_answer` может быть неточным | RAG-контур с индексом по документации технологий |
| Нет актуальной документации | LLM может генерировать устаревшие ответы о версиях, API | Интеграция с актуальной документацией через retrieval |
| Потеря ранних ходов у Interviewer | При длинных сессиях ранние ходы выпадают из `HISTORY_WINDOW_TURNS` | Суммаризация ранних ходов (sliding window + summary) |
| Полная история у Evaluator | Для длинных сессий контекст может быть велик | Map-reduce суммаризация перед финальной оценкой |
| Нет кросс-сессионной памяти | Каждая сессия независима, нет профиля кандидата | Персистентный профиль кандидата с историей сессий |

---

## 6. Точки расширения для RAG

Если в будущем потребуется классический retrieval-контур, рекомендуемые точки интеграции:

1. **ObserverAgent** — подключение RAG для верификации фактов кандидата. Источник: индекс по официальной документации технологий.
2. **InterviewerAgent** — подключение RAG для генерации вопросов на основе базы знаний. Источник: индекс по типовым вопросам для интервью.
3. **EvaluatorAgent** — подключение RAG для калибровки оценки. Источник: индекс по критериям грейдов и ожиданиям к уровням.

Интеграция рекомендуется через создание нового компонента `RetrieverService` в `src/app/retrieval/`, который будет вызываться из `_build_*_context()` методов агентов, не изменяя интерфейс `BaseAgent.process()`.
